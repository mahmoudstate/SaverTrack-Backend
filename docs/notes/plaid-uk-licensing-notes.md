## 1. الترخيص (Licensing)
- التطبيق يحتاج ترخيص **AIS (Account Information Services)** في UK لأنه يعرض بيانات بنكية مجمّعة للمستخدم النهائي.
- مسارين:
  1. **ترخيص FCA مستقل (own AISP authorisation)** — وقت أطول.
  2. **العمل كـ Agent تحت ترخيص Plaid كـ AISP** — أسرع للسوق، لكن يتطلب موافقة/مراجعة (approval & compliance process) من Plaid قبل go-live.
- تسعير تقريبي لمسار الـ Agency (Ballpark فقط، غير نهائي):
  - **~£15,000** رسوم onboarding لمرة واحدة
  - **~£1,000/شهر** رسوم منصة مستمرة (agency platform fee)
  - يخضع للمراجعة النهائية والـ scoping.
- تسعير UK/EU بشكل عام: **عقد مخصص (custom contract)**، لا يوجد self-serve / pay-as-you-go.

## 2. Recurring Transactions (UK)
- الـ endpoint `/transactions/recurring/get` **مدعوم في UK** كإضافة (add-on) للمؤسسات التي تدعم Transactions.
- يُنصح بطلب **180 يوم على الأقل** من تاريخ المعاملات قبل استدعاء recurring لأفضل نتائج.
- لا يوجد ضمان تطابق تام مع بيانات US Sandbox — يعتمد على المؤسسة المستهدفة.

## 3. إعادة الموافقة (Re-consent / Re-authentication)
- الموافقة تُجدَّد كل **90 يوم** في UK.
- التجديد يتم عادة كـ "re-consent" بسيط عبر Plaid، وليس إعادة مصادقة كاملة مع البنك.
- Plaid تتبع موعد الانتهاء وتُنبّه المستخدم قبله عبر **Link update mode**.
- إذا انتهت الموافقة: الوصول للبيانات يتوقف حتى يعيد المستخدم التأكيد.

## 4. تاريخ المعاملات عند أول Pull
- أقصى مدى ممكن طلبه: **24 شهر**.
- الافتراضي (لو معملتش تحديد): **90 يوم**.
- أول مزامنة عادة على مراحل: بيانات حديثة أولاً، ثم backfill تاريخي.

## 5. أرصدة بطاقات الائتمان
- لو البنك يوفر available balance → Plaid ترجعها.
- لو مش متوفرة → ترجع `null`.
- للحسابات الائتمانية: `available` عادة = الحد الائتماني − الرصيد الحالي (معدّلة حسب المعاملات المعلقة)، لكن ليست كل المؤسسات تحسبها بنفس الطريقة.

## 6. هيكل التسعير العام
- UK/EU: عقد مخصص فقط.
- Transactions و Recurring Transactions و Balance — كل منها يُحاسب بشكل منفصل.
- الهيكل التجاري عادة: platform fee + usage commitment، وليس سعر بالطلب فقط.
